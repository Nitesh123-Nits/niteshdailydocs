    üß† 1. ‚ÄúHot Key‚Äù ‚Äî Meaning & Context

    Definition:
    A hot key refers to a specific key or item that receives disproportionately high access traffic compared to others in a distributed system (cache, DB, or storage).

    üîç Example:

    • Suppose you have a Redis cache storing product details:

      product:{id} -> productDetails

      If product:123 is an iPhone 16, and 80% of your users view that product page, then this key (product:123) becomes a hot key.

    ‚ö†Ô∏è Problem:

    • That single key gets too many reads/writes on one cache node or partition ‚Üí hotspotting, load imbalance, or even throttling.

    • It breaks horizontal scalability assumptions (since load is uneven).

    üõ†Ô∏è Solutions:

    • Sharding or Key Hashing: Spread requests across nodes by random suffixing keys (e.g., product:123:1, product:123:2) and combine results.

    • Caching at Edge/CDN: Offload popular key access closer to users.

    • Replication: Store replicas for read-heavy keys.

    • Async batching: Debounce or batch reads/writes on the same key.

    üí¨ When to use the word:

    ‚ÄúWe noticed a hot key pattern in Redis due to one product‚Äôs extreme popularity, which caused uneven load. We mitigated it by adding key randomization and read replicas.‚Äù

    That sounds very seasoned and real-world.

    üåê 2. ‚ÄúHot API‚Äù ‚Äî Meaning & Context

    Definition:
    A hot API (or hot endpoint) is an API endpoint that receives a very high QPS (queries per second) compared to other endpoints ‚Äî becoming a bottleneck or hotspot in your microservice architecture.

    üîç Example:

    • In an e-commerce platform:

      • /api/products/{id} might be hit 100x more than /api/users/{id}

      • /api/search or /api/homepage could be hot APIs since they‚Äôre called on every page load.

    ‚ö†Ô∏è Problem:

    • These endpoints experience latency spikes, rate-limit issues, or backend contention.

    • Could trigger DB load spikes or cache thrashing.

    üõ†Ô∏è Solutions:

    • Cache responses (Redis, CDN, in-memory).

    • Rate limiting and load shedding.

    • Read replicas behind hot APIs.

    • Async pipelines or CQRS (split read/write).

    • Edge caching or API Gateway-level caching.

    üí¨ When to use the word:

    ‚ÄúThe /api/homepage endpoint turned into a hot API after we went live ‚Äî so we cached its response in Redis for 2 seconds to flatten load spikes.‚Äù

    That phrasing gives instant credibility.

    üßä 3. ‚ÄúCold Key‚Äù or ‚ÄúCold Path‚Äù

    Cold key = rarely accessed data.
    Cold path = code path not frequently used (opposite of hot path).

    Example:

    • In analytics systems, today‚Äôs data is hot, last year‚Äôs is cold.
      ‚Üí Hot data lives in Redis or SSDs; cold data in S3 or archive storage.

    üí¨ Use case:

    ‚ÄúWe maintain a hot-cold data split where frequently accessed metrics stay in Redis and older metrics are moved to S3.‚Äù

    ‚ö° 4. ‚ÄúHot Partition‚Äù or ‚ÄúHot Shard‚Äù

    If the load imbalance happens at the partition level (in Kafka, Cassandra, MongoDB, etc.), that‚Äôs called a hot partition or hot shard.

    Example:

    • Kafka topic has 10 partitions.

    • Partition #3 gets 90% of writes ‚Üí consumers on that partition are overwhelmed.

    üí¨ Use case:

    ‚ÄúWe faced a hot partition issue in Kafka because our partitioning key wasn‚Äôt uniform ‚Äî we fixed it by using a composite key with random salt.‚Äù

    üî• 5. ‚ÄúHot Path‚Äù or ‚ÄúHot Code Path‚Äù

    Hot path = the most frequently executed part of code in a request flow or microservice.

    Example:

    • In a service, the main request handler and database query path are ‚Äúhot paths.‚Äù

    • Profiling and optimizing hot paths lead to big performance gains.

    üí¨ Use case:

    ‚ÄúWe optimized the hot path by switching from JSON to binary serialization, cutting 30% CPU overhead.‚Äù

    ‚öôÔ∏è 1. Cache Stampede (a.k.a. Dogpile Effect)

    üß† Concept:

    A cache stampede occurs when a cache entry expires, and suddenly multiple concurrent requests try to fetch and recompute the same data ‚Äî overloading the database or backend.

    It‚Äôs like everyone rushing to refill the same cache entry at once.

    üö® Problem It Solves:

    Without protection, the system experiences a sudden traffic spike to the origin (DB, microservice, etc.), leading to latency or even cascading failures.

    üí° Common Mitigation Strategies:

    • Locking / Single Flight: Only one thread recomputes the data; others wait.

    • Early Revalidation (Soft TTL): Refresh cache before it expires.

    • Randomized TTLs: Add jitter to cache expiry times.

    • Background Refresh: Asynchronously update cache in the background.

    üí¨ Real-world Example:

    ‚ÄúWe faced a cache stampede issue in Redis when our homepage cache expired, leading to thousands of DB hits in seconds. We implemented soft TTL and added random jitter to prevent synchronized expirations.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúHow is it different from a thundering herd?‚Äù
      ‚Üí A cache stampede is specifically about cache expiration, while thundering herd is a general traffic surge to any resource.

    • ‚ÄúHow do you prevent it without adding latency?‚Äù
      ‚Üí Use soft TTL or background refresh ‚Äî refresh in advance.

    üêò 2. Thundering Herd Problem

    üß† Concept:

    A thundering herd happens when many clients or threads wake up simultaneously to process the same event or access a single shared resource ‚Äî causing a massive load surge.

    It‚Äôs a synchronized spike problem, not necessarily cache-specific.

    üö® Problem It Solves:

    Avoids resource contention and spike overload on downstream services, databases, or locks.

    üí° Common Mitigation Strategies:

    • Request coalescing (merge identical requests)

    • Exponential backoff + jitter on retries

    • Distributed rate limiting

    • Batching or queueing incoming requests

    • Staggered wake-ups (using randomized timers)

    üí¨ Real-world Example:

    ‚ÄúWe noticed a thundering herd on our Kafka consumers when a topic became available ‚Äî all consumers woke up simultaneously. We fixed it by adding jittered scheduling and batch consumption.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúWhat‚Äôs the difference between thundering herd and cache stampede?‚Äù
      ‚Üí Cache stampede is a special case of thundering herd.

    • ‚ÄúHow would you handle it in a distributed queue system?‚Äù
      ‚Üí Use staggered polling intervals, exponential backoff, or leader election for coordination.

    üßµ 3. Backpressure

    üß† Concept:

    Backpressure is a flow control mechanism that helps systems slow down producers when consumers are overwhelmed ‚Äî preventing queue buildup, OOM errors, or latency spikes.

    Think of it like a pressure valve in a water pipe ‚Äî it signals upstream systems to stop or slow data flow.

    üö® Problem It Solves:

    In event-driven or streaming systems, producers may emit messages faster than consumers can handle. Without backpressure, you get queue overflow or system crashes.

    üí° Common Mitigation Strategies:

    • Reactive Streams / Flow control signals (e.g., in Spring WebFlux, Kafka, gRPC streaming)

    • Bounded queues

    • Dropping or buffering strategies

    • Load shedding when thresholds are reached

    üí¨ Real-world Example:

    ‚ÄúWe implemented backpressure in our reactive pipeline to avoid overloading the downstream payment service when the Kafka topic spiked.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúWhat happens if backpressure isn‚Äôt implemented?‚Äù
      ‚Üí Consumers get overwhelmed, causing latency, memory buildup, or service crashes.

    • ‚ÄúHow does backpressure differ from rate limiting?‚Äù
      ‚Üí Rate limiting controls external clients; backpressure controls internal system flow.

    ‚ö° 4. Circuit Breaker Pattern

    üß† Concept:

    The Circuit Breaker is a resilience pattern that prevents cascading failures by cutting off requests to a failing downstream service.

    It‚Äôs inspired by electrical circuits ‚Äî once too many failures occur, the breaker opens and blocks further calls temporarily.

    ‚öôÔ∏è States:

    1. Closed: Normal operation.

    2. Open: Block requests due to repeated failures.

    3. Half-Open: Allow limited requests to test recovery.

    üö® Problem It Solves:

    Prevents overloading a service that‚Äôs already failing, and stops consuming resources on guaranteed failures.

    üí° Common Implementations:

    • Netflix Hystrix (legacy)

    • Resilience4j (modern Java library)

    • Istio / Envoy sidecars for service mesh

    üí¨ Real-world Example:

    ‚ÄúWe applied a circuit breaker around our user-service calls to prevent cascading failures when it went down. The breaker opened after 5 timeouts, and half-opened after 30 seconds.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúHow is it different from retry?‚Äù
      ‚Üí Retry keeps trying; circuit breaker stops trying after a threshold.

    • ‚ÄúWhen do you reset the circuit breaker?‚Äù
      ‚Üí After a cooldown period or during half-open state tests.

    • ‚ÄúWhat metrics decide opening the breaker?‚Äù
      ‚Üí Failure rate, timeout rate, or latency thresholds.

    ü™£ 5. Bulkhead Pattern

    üß† Concept:

    Bulkhead isolation means dividing a system into isolated compartments, so if one fails, it doesn‚Äôt sink the entire system ‚Äî just like watertight sections in a ship.

    üö® Problem It Solves:

    Prevents cascading failures by isolating resource usage between services or threads.

    üí° Common Mitigation Strategies:

    • Separate thread pools, connection pools, or service quotas.

    • Limit impact radius of failure.

    üí¨ Real-world Example:

    ‚ÄúWe used bulkhead isolation by assigning dedicated thread pools per downstream dependency to prevent one slow API from starving others.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúHow does this differ from circuit breaker?‚Äù
      ‚Üí Circuit breaker stops calling a failing service; bulkhead prevents one from impacting others.

    üöß 6. Rate Limiting

    üß† Concept:

    Rate limiting restricts how many requests a client or service can make in a given time window.

    üö® Problem It Solves:

    Protects services from abuse, DDoS attacks, or unintentional client spikes.

    üí° Common Algorithms:

    • Token Bucket

    • Leaky Bucket

    • Fixed Window / Sliding Window

    üí¨ Real-world Example:

    ‚ÄúWe applied rate limiting using a token bucket algorithm at API Gateway level to prevent clients from overwhelming our backend.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúHow does it differ from backpressure?‚Äù
      ‚Üí Rate limiting controls incoming requests from outside; backpressure controls internal flow.

    üí• 7. Load Shedding

    üß† Concept:

    Load shedding is the deliberate dropping of requests when the system is near capacity ‚Äî to keep response times stable and avoid total collapse.

    üö® Problem It Solves:

    Maintains system stability under extreme load.

    üí° Common Mitigation Strategies:

    • Prioritize high-value requests.

    • Drop non-essential background tasks.

    • Return HTTP 503 with retry hints.

    üí¨ Real-world Example:

    ‚ÄúDuring peak load, we applied load shedding by rejecting low-priority analytics events to maintain latency SLAs on user transactions.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúIsn‚Äôt dropping requests bad?‚Äù
      ‚Üí Sometimes it‚Äôs better to fail fast for some users than to fail completely for all.

    ‚è±Ô∏è 8. Retry with Exponential Backoff & Jitter

    üß† Concept:

    Instead of retrying failed requests immediately (which worsens overload), use exponential backoff ‚Äî wait progressively longer before retrying.

    Jitter adds randomness to prevent synchronized retries from many clients (which causes thundering herd).

    üí¨ Real-world Example:

    ‚ÄúWe used exponential backoff with jitter in our gRPC clients to avoid synchronized retry storms when the downstream service recovered.‚Äù

    üéØ Interview Cross-Questions:

    • ‚ÄúWhy add jitter?‚Äù
      ‚Üí To desynchronize retries ‚Äî otherwise, everyone retries at the same time.

    ‚ö° Advanced System Design Concepts ‚Äì Block 2

    üß© 1. Idempotency

    üß† Concept:

    An idempotent operation is one that can be safely retried multiple times without changing the result beyond the initial application.

    In distributed systems, retries are common due to transient failures ‚Äî idempotency ensures correctness even under duplicate requests.

    üí° Example:

    • PUT /users/123 with a full body is idempotent (same result no matter how many times it‚Äôs called).

    • POST /users (creating new resource) is non-idempotent, unless you use idempotency keys.

    ‚öôÔ∏è Techniques:

    • Use unique request IDs (UUID or idempotency key) to detect duplicates.

    • Maintain a request log table to discard duplicate operations.

    üí¨ Interview Insight:

    ‚ÄúIn our payment microservice, we made transaction creation idempotent using a unique idempotency key to avoid double charges on retries.‚Äù

    Cross-question:

    • How is it different from at-least-once vs exactly-once delivery?
      ‚Üí Idempotency is an application-level guarantee; those are message delivery semantics.

    ‚öñÔ∏è 2. Consistency Models

    üß† Concept:

    Distributed systems often sacrifice immediate consistency to gain availability and partition tolerance (CAP theorem).
    There are multiple consistency models, each balancing latency vs correctness.

    üí° Common Models:

    • Strong Consistency: Always up-to-date (e.g., traditional RDBMS).

    • Eventual Consistency: Updates propagate asynchronously (e.g., DynamoDB, Cassandra).

    • Read-Your-Writes: You‚Äôll always see your own recent writes.

    • Monotonic Reads: Once you‚Äôve seen a value, you‚Äôll never see an older one.

    üí¨ Interview Insight:

    ‚ÄúWe chose eventual consistency for user activity feeds since real-time strict consistency wasn‚Äôt critical, but latency was.‚Äù

    Cross-Question:

    • When do you prefer strong consistency?
      ‚Üí For transactions like payments or inventory where correctness matters more than speed.

    üßÆ 3. CAP Theorem & PACELC Theorem

    üß† Concept:

    • CAP Theorem: A distributed system can only guarantee two of three ‚Äî Consistency, Availability, Partition Tolerance.

    • PACELC Theorem extends it: Even when there‚Äôs no partition, you must choose between Latency (L) and Consistency (C).

    üí¨ Interview Insight:

    ‚ÄúWe designed our system as AP ‚Äî prioritizing availability over consistency during partitions, acceptable for analytics data.‚Äù

    Cross-Question:

    • What would you pick for a payment system?
      ‚Üí CP ‚Äî Consistency + Partition tolerance, sacrificing availability.

    üì¶ 4. Eventual Consistency & Conflict Resolution

    üß† Concept:

    In eventually consistent systems, replicas will converge over time. But if two replicas accept writes concurrently, conflicts occur.

    üí° Resolution Strategies:

    • Last Write Wins (LWW)

    • Vector Clocks

    • CRDTs (Conflict-Free Replicated Data Types)

    üí¨ Interview Insight:

    ‚ÄúWe used CRDTs in our chat service to merge concurrent edits without conflict ‚Äî it guarantees eventual convergence.‚Äù

    Cross-Question:

    • When would you prefer LWW over CRDT?
      ‚Üí When order matters more than intermediate states (like timestamps in logs).

    üï∏Ô∏è 5. Data Sharding & Hotspotting

    üß† Concept:

    Sharding splits large datasets horizontally into shards (smaller, distributed partitions).
    Hotspotting occurs when uneven data access overloads one shard (similar to hot partition).

    üí° Sharding Strategies:

    • Hash-based (uniform distribution)

    • Range-based (sorted order)

    • Directory-based (lookup table)

    üí¨ Interview Insight:

    ‚ÄúWe implemented hash-based sharding to evenly distribute users across shards and avoid hotspotting.‚Äù

    Cross-Question:

    • How do you rebalance shards dynamically?
      ‚Üí Add new shards, migrate subsets, or use consistent hashing.

    üï∞Ô∏è 6. Leader Election

    üß† Concept:

    In distributed coordination, leader election ensures that only one node performs a critical task at a time (avoiding race conditions).
